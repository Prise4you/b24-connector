# Коннектор Bitrix24 → база знаний (NotebookLM + нативный BitrixGPT)

Пилот: выгружает данные одного проекта Bitrix24 (задачи + комментарии, чат, файлы) в
агрегированные документы и заливает их в NotebookLM и/или на Диск Bitrix24 (для нативного
агента BitrixGPT). Автопополнение — через offline-события (без публичного сервера).

## Что нужно сделать в портале (один раз)

### 1. Создать входящий вебхук
Bitrix24 → **Разработчикам** → **Другое** → **Входящий вебхук** → **Создать**.

Отметить права (scope):
- `task` — задачи и комментарии
- `im` — чаты (комментарии задач и чат группы читаются через im)
- `disk` — файлы
- `imopenlines` — открытые линии (если нужны)
- `crm` — сущности CRM и лента (если нужны)
- `event` — для автопополнения (offline-события)

Скопировать URL вида:
`https://ВАШ-ПОРТАЛ.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/`

> ⚠️ Этот URL = ключ доступа. Не публиковать, не коммитить. Кладётся только в `config.json` (он в .gitignore).

### 2. Узнать ID тестового проекта (рабочей группы)
Открыть нужную рабочую группу/проект → в адресной строке `/workgroups/group/<ID>/` → это `group_id`.

### 3. Сообщить ассистенту
- URL вебхука (или сразу вписать в `config.json`)
- `group_id` тестового проекта
- Версию портала и тариф (для нативного RAG — нужен 24.700.0+ и тариф с базами знаний)

## Настройка

```bash
cp config.example.json config.json
# вписать webhook_url и group_id
```

Проверить доступ:
```bash
python3 b24_client.py "https://ВАШ-ПОРТАЛ.bitrix24.ru/rest/1/ТОКЕН/"
# → OK, авторизован как: Имя Фамилия ID N
```

## Запуск

```bash
python3 connector.py --dry-run     # собрать документы в out/, без загрузки
python3 connector.py               # собрать + залить в NotebookLM
python3 connector.py --bind        # зарегистрировать offline-события (один раз)
python3 connector.py --poll        # забрать события; пересобрать только при изменениях
```

Автопополнение по расписанию (launchd, каждые 10 мин):
```bash
bash launchd/install.sh            # bind + установка launchd-агента
```

## Безопасность (Продукт A — внутренний инструмент)

- **Фильтр секретов** (`security.py`): перед отправкой в RAG маскируются passwork-ссылки,
  токены, пароли, bearer/basic-auth, SSH/PEM-ключи, Bitrix-вебхуки, карты, паспорта, СНИЛС,
  телефоны, email. Аудит — `logs/security_audit.jsonl`.
- **Admin-check**: коннектор требует прав администратора портала (`user.admin`).
- **ПДн в открытых линиях/CRM**: сама переписка/контент — разрешена к выгрузке в NotebookLM
  (Google = трансграничная передача, 152-ФЗ ст. 12 — АНИТ как оператор берёт на себя это
  решение для внутреннего использования). Жёстко фильтруется только то, что реально
  идентифицирует человека или даёт доступ: телефоны, email, пароли/токены/доступы —
  см. список фильтра секретов выше. Технического блока «нельзя включить оба таргета
  одновременно» в коде нет и не планируется — это осознанное решение (2026-07-17), не баг.
- `config.json` → права 600. Логи и `out/` — в `.gitignore`.
